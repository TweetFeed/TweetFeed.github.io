What you can hunt

1. Quick start - get raw IOCs

Four ways to ingest the feed. Pick the one that matches your downstream tool.

#1 REST API #2 CSV download #3 RSS #4 MISP feed

All IOCs from today as JSON:

curl -s https://api.tweetfeed.live/v1/today | jq .

Filter by IOC type (url, domain, ip, sha256, md5):

curl -s https://api.tweetfeed.live/v1/today/url

Combine type and tag (this week's CobaltStrike IPs):

curl -s https://api.tweetfeed.live/v1/week/ip/CobaltStrike

Response shape (one IOC per array element):

{
  "date": "2026-05-03 06:07:00",
  "user": "PhishStats",
  "type": "domain",
  "value": "evil.example.shop",
  "tags": ["#phishing"],
  "tweet": "https://x.com/PhishStats/status/..."
}

CSVs cover four windows: today, week, month, year. Header: date,user,type,value,tags,tweet.

wget https://raw.githubusercontent.com/0xDanielLopez/TweetFeed/master/today.csv
wget https://raw.githubusercontent.com/0xDanielLopez/TweetFeed/master/week.csv
wget https://raw.githubusercontent.com/0xDanielLopez/TweetFeed/master/month.csv

Quick filter without jq:

awk -F, '$3=="\"ip\""' week.csv | cut -d, -f4

RSS feed with the latest IOCs (refreshed every 15 min):

curl -s https://raw.githubusercontent.com/0xDanielLopez/TweetFeed/master/rss.xml

Python consumer using feedparser:

import feedparser
feed = feedparser.parse("https://raw.githubusercontent.com/0xDanielLopez/TweetFeed/master/rss.xml")
for entry in feed.entries:
    # entry.title is the raw IOC value
    # entry.link is the source tweet URL
    print(entry.title, entry.link)

TweetFeed publishes a MISP-format manifest with 4 Events (today / week / month / year). Add it as a feed:

Sync Actions → Feeds → Add Feed:

Source format: misp
Provider:      TweetFeed
Auto-pull:     yes
Default tags:  TweetFeed, tlp:clear, type:OSINT

2. Terminal & shell

Six recipes you can paste into a shell right now. All return clean stdout suitable for piping into your tooling.

2.1 Daily blocklist refresh - cron-friendly, runs every 15 min and updates a flat file:

*/15 * * * * curl -sfL https://api.tweetfeed.live/v1/today/url \
  | jq -r '.[].value' | sort -u > /etc/blocklists/tweetfeed-urls.txt

2.2 Slice by malware family tag - only IPs tagged Cobalt Strike from this week:

curl -sL https://api.tweetfeed.live/v1/week/ip/CobaltStrike | jq -r '.[].value'

2.3 Pivot from a researcher - every IOC published this week by @JAMESWT_MHT (filter is client-side because the REST endpoint does not accept user as a path filter):

curl -sL https://api.tweetfeed.live/v1/week \
  | jq -r '.[] | select(.user=="JAMESWT_MHT") | "\(.type)\t\(.value)\t\(.tags|join(","))"'

2.4 Whois ownership of all malicious IPs from today:

curl -sL https://api.tweetfeed.live/v1/today/ip | jq -r '.[].value' \
  | xargs -I{} sh -c 'echo "=== {} ==="; whois {} | grep -iE "orgname|country"'

2.5 Resolve and cluster by IP - find phishing domains sharing the same upstream infrastructure:

curl -sL https://api.tweetfeed.live/v1/today/domain | jq -r '.[].value' \
  | while read d; do printf "%s\t" "$d"; dig +short "$d" | head -1; done \
  | sort -k2 | awk '$2!=""'

2.6 IOC to source tweet - every IOC carries provenance, useful before you alert on it:

curl -sL https://api.tweetfeed.live/v1/today \
  | jq -r '.[] | select(.value=="evil.example.com") | .tweet'

3. SIEM playbooks

Use TweetFeed as a lookup table or external source against your log indexes.

#1 Splunk SPL #2 Sentinel KQL #3 Defender (KQL) #4 Elastic #5 Chronicle YARA-L

Setup - schedule a saved search to refresh the lookup table from the TweetFeed CSV every 15 min (savedsearches.conf):

[TweetFeed_today_lookup_refresh]
search = | inputlookup append=t external_lookup tweetfeed_today.csv \
         | outputlookup tweetfeed_today.csv
cron_schedule = */15 * * * *
dispatch.earliest_time = -15m
enableSched = 1

3.1 Match outbound proxy logs vs TweetFeed URLs and domains:

index=proxy [
  | inputlookup tweetfeed_today.csv
  | where type IN ("url","domain")
  | rename value AS url
  | fields url
] | stats count, values(tags), values(tweet) by src_ip, url

3.2 DNS query enrichment - any DNS query that hits a TweetFeed domain in the last 7d:

index=dns sourcetype=dns:query [
  | inputlookup tweetfeed_week.csv
  | where type="domain" | rename value AS query | fields query
] | table _time, src_ip, query, [tweetfeed.tags], [tweetfeed.tweet]

3.3 Real-time alert - save the previous query as a Notable Event with severity=medium and trigger immediately when count > 0.

3.4 externaldata against today's CSV - DNS hits in last 24h:

let TweetFeedDomains = externaldata(date:string, user:string, type:string, value:string, tags:string, tweet:string)
  [@"https://raw.githubusercontent.com/0xDanielLopez/TweetFeed/master/today.csv"]
  with (format="csv", ignoreFirstRecord=true)
  | where type == "domain"
  | project Domain = value, Tags = tags, Tweet = tweet;
DnsEvents
| where TimeGenerated > ago(24h)
| where Name in (TweetFeedDomains | project Domain)
| join kind=leftouter TweetFeedDomains on $left.Name == $right.Domain
| project TimeGenerated, Computer, ClientIP, Name, Tags, Tweet

3.5 SecurityEvent against TweetFeed IPs - logon source pivot against week.csv:

let TweetFeedIPs = externaldata(date:string, user:string, type:string, value:string, tags:string, tweet:string)
  [@"https://raw.githubusercontent.com/0xDanielLopez/TweetFeed/master/week.csv"]
  with (format="csv", ignoreFirstRecord=true)
  | where type == "ip"
  | project IPAddress = value, Tags = tags, Tweet = tweet;
SecurityEvent
| where TimeGenerated > ago(7d)
| where IpAddress in (TweetFeedIPs | project IPAddress)
| join kind=leftouter TweetFeedIPs on $left.IpAddress == $right.IPAddress
| project TimeGenerated, Computer, Account, IpAddress, Tags, Tweet

Defender for Endpoint advanced hunting (KQL). The 4 sub-tabs cover every IOC type. Swap week.csv for today.csv or month.csv depending on your freshness vs coverage trade-off.

3.7 Chronicle YARA-L - DNS lookup matching a reference list of TweetFeed domains:

rule tweetfeed_domain_lookup {
  meta:
    description = "DNS query to a TweetFeed-listed domain"
    severity = "Medium"
  events:
    $dns.metadata.event_type = "NETWORK_DNS"
    $dns.network.dns.questions.name = $domain
  match:
    $domain over 5m
  condition:
    $dns and $domain in %tweetfeed_domains
}

Populate %tweetfeed_domains reference list from /v1/week/domain on a 15-min cadence.

4. EDR / XDR

Push TweetFeed IOCs into the EDR's IOC management plane, or hunt your endpoint telemetry for matches.

#1 CrowdStrike Falcon #2 SentinelOne #3 Wazuh

4.1 Push hashes as IOCs via the IOC Management API. Falcon's IOC API accepts md5, sha256, ipv4, ipv6 and domain types:

TOKEN=$(curl -sX POST "https://api.crowdstrike.com/oauth2/token" \
  -d "client_id=$CS_ID&client_secret=$CS_SECRET" | jq -r .access_token)

curl -sL https://api.tweetfeed.live/v1/today/sha256 \
  | jq '[.[] | {type:"sha256", value:.value, action:"detect",
                platforms:["windows","mac","linux"],
                source:"tweetfeed.live",
                description:("tags: " + (.tags|join(",")))}]' \
  | jq '{indicators: .}' \
  | curl -sX POST "https://api.crowdstrike.com/iocs/entities/indicators/v1" \
      -H "Authorization: Bearer $TOKEN" -H "Content-Type: application/json" \
      -d @-

4.2 Falcon Insight Search - DNS lookups to TweetFeed domains in the last 7d (paste into the Falcon UI as an Event Search):

event_simpleName=DnsRequest
| join domains_today=[
    !curl https://api.tweetfeed.live/v1/today/domain | .[].value
  ] DomainName
| table @timestamp, ComputerName, DomainName, ContextProcessId

Pattern shown above; in real Falcon LogScale you'll typically pre-stage the domain list in a lookup file or function.

4.3 Process executed a TweetFeed-flagged binary (hash match):

event_simpleName=ProcessRollup2 SHA256HashData IN [list_of_tweetfeed_sha256]
| table @timestamp, ComputerName, FileName, CommandLine, ParentBaseFileName

4.4 Deep Visibility query - file SHA256 matches TweetFeed hash list (paste into Singularity's Deep Visibility):

SrcProcSha256 IN ("hash1","hash2",...) OR TgtFileSha256 IN ("hash1","hash2",...)

4.5 Bulk import via Singularity API:

curl -sL https://api.tweetfeed.live/v1/today/sha256 \
  | jq '[.[] | {value:.value, type:"SHA256", source:"TweetFeed",
                description:("tags: " + (.tags|join(",")))}]' \
  | jq '{data: ., filter: {tenant: true}}' \
  | curl -sX POST "https://YOUR_MGMT.sentinelone.net/web/api/v2.1/threat-intelligence/iocs" \
      -H "Authorization: ApiToken $S1_TOKEN" -H "Content-Type: application/json" \
      -d @-

4.6 CDB list config - refresh a CDB list from TweetFeed and reference it in a custom rule. Save the script and run via cron:

#!/bin/bash
curl -sfL https://api.tweetfeed.live/v1/today/domain \
  | jq -r '.[].value + ":TweetFeed"' \
  > /var/ossec/etc/lists/tweetfeed-domains

Add to /var/ossec/etc/rules/local_rules.xml:

<group name="tweetfeed,">
  <rule id="100100" level="10">
    <list field="dns_query" lookup="match_key">etc/lists/tweetfeed-domains</list>
    <description>TweetFeed-listed domain queried via DNS</description>
  </rule>
</group>

5. Network & firewall

Stop traffic to TweetFeed-listed indicators at the perimeter.

5.1 Suricata DNS rules generator - emit one alert rule per malicious domain (assigns SIDs starting at 2,000,000 to avoid collision with ET rules):

curl -sL https://api.tweetfeed.live/v1/today/domain | jq -r '.[].value' \
  | awk 'BEGIN{n=2000000} {printf "alert dns any any -> any any (msg:\"TweetFeed bad domain %s\"; dns_query; content:\"%s\"; nocase; sid:%d; rev:1;)\n", $1, $1, n+NR}' \
  > /etc/suricata/rules/tweetfeed.rules
suricatasc -c reload-rules

5.2 Pi-hole sinkhole - script wrapper to keep tweetfeed.list in sync. Add it as a custom adlist in Pi-hole admin:

# /usr/local/bin/refresh-tweetfeed-pihole.sh
curl -sfL https://api.tweetfeed.live/v1/today/domain \
  | jq -r '.[].value' | sort -u \
  > /etc/pihole/tweetfeed.list
pihole -g  # regenerate gravity

5.3 pfSense / OPNsense alias - host a pre-processed plain-text IP list on any internal HTTP server, then import as URL Table alias. Firewall → Aliases → URL Table (IPs):

curl -sfL https://api.tweetfeed.live/v1/today/ip | jq -r '.[].value' > /var/www/html/tweetfeed-ip.txt
Name:           tweetfeed_ip_today
Type:           URL Table (IPs)
URL:            http://internal.example.com/tweetfeed-ip.txt
Update Freq:    1 day

5.4 Squid proxy ACL - block HTTP/HTTPS connections to TweetFeed-listed domains:

curl -sfL https://api.tweetfeed.live/v1/today/domain | jq -r '.[].value' > /etc/squid/tweetfeed.txt
echo 'acl tweetfeed_bad dstdomain "/etc/squid/tweetfeed.txt"' >> /etc/squid/squid.conf
echo 'http_access deny tweetfeed_bad' >> /etc/squid/squid.conf
squid -k reconfigure

6. TIP & SOAR

Ingest TweetFeed into your Threat Intelligence platform or pivot from cases.

6.1 MISP feed install - TweetFeed publishes 4 MISP-format Events (today / week / month / year) regenerated every 15 min. Sync Actions → Feeds → Add Feed:

URL:           https://raw.githubusercontent.com/0xDanielLopez/TweetFeed/master/misp/manifest.json
Source format: misp
Provider:      TweetFeed
Auto-pull:     yes
Distribution:  Your organisation only (or as needed)
Default tags:  TweetFeed, tlp:clear, type:OSINT

After saving, click Fetch and store all feed data to seed the index, then enable Cron Jobs → Feed Pull for ongoing auto-refresh.

6.2 TheHive bulk observable import - add today's IOCs to a case using thehive4py:

import requests
from thehive4py import TheHiveApi
from thehive4py.models import CaseObservable

api = TheHiveApi("https://thehive.example.com", "$THEHIVE_API_KEY")
case_id = "~12345"  # your case
type_map = {"url":"url","domain":"domain","ip":"ip","md5":"hash","sha256":"hash"}

iocs = requests.get("https://api.tweetfeed.live/v1/today").json()
for ioc in iocs:
    obs = CaseObservable(
        dataType=type_map[ioc["type"]],
        data=ioc["value"],
        tags=["TweetFeed", *ioc["tags"]],
        message=f"Source: {ioc['tweet']}",
        tlp=2, ioc=True)
    api.create_case_observable(case_id, obs)

7. Detection content

Generate detection-as-code artifacts (Sigma, YARA, STIX) from the feed for your detection pipeline.

7.1 Sigma rule template - DNS lookup to a TweetFeed-listed domain. Populate %TweetFeedDomains% via your Sigma processing pipeline (e.g. sigmac with a lookup expansion):

title: TweetFeed-listed Malicious Domain Resolution
id: f1c0c0d0-1111-4444-9999-aaaaaaaaaaaa
status: experimental
description: Detects DNS lookups to domains published by TweetFeed within the last week
references:
  - https://tweetfeed.live/
  - https://api.tweetfeed.live/v1/week/domain
author: TweetFeed
date: 2026/05/03
logsource:
  product: windows
  service: sysmon
detection:
  selection:
    EventID: 22
    QueryName|expand: '%TweetFeedDomains%'
  condition: selection
falsepositives:
  - Allowlisted internal infrastructure
level: medium
tags:
  - attack.command_and_control
  - attack.t1071.004

7.2 YARA rule generated from the SHA256 feed (one rule per hash; collapse into a master rule with multiple hash.sha256(...) conditions if you want a single match):

import requests
iocs = requests.get("https://api.tweetfeed.live/v1/week/sha256").json()
print('import "hash"\n')
for ioc in iocs:
    name = "tweetfeed_" + ioc["value"][:12]
    tags = " ".join(t.lstrip("#") for t in ioc.get("tags", []))
    print(f'rule {name} : tweetfeed {tags} {{')
    print(f'  meta:')
    print(f'    description = "TweetFeed sha256 ({tags})"')
    print(f'    source = "{ioc["tweet"]}"')
    print(f'  condition:')
    print(f'    hash.sha256(0, filesize) == "{ioc["value"]}"')
    print(f'}}\n')

7.3 STIX 2.1 export - convert the JSON feed into Indicator SDOs and emit a Bundle:

import requests
from stix2 import Indicator, Bundle

PATTERN = {
  "url":    lambda v: f"[url:value = '{v}']",
  "domain": lambda v: f"[domain-name:value = '{v}']",
  "ip":     lambda v: f"[ipv4-addr:value = '{v}']",
  "sha256": lambda v: f"[file:hashes.'SHA-256' = '{v}']",
  "md5":    lambda v: f"[file:hashes.MD5 = '{v}']",
}

iocs = requests.get("https://api.tweetfeed.live/v1/today").json()
indicators = [
    Indicator(
        name=ioc["value"],
        pattern=PATTERN[ioc["type"]](ioc["value"]),
        pattern_type="stix",
        labels=ioc["tags"],
        description=f"Source: {ioc['tweet']}",
    )
    for ioc in iocs if ioc["type"] in PATTERN
]
print(Bundle(objects=indicators).serialize(pretty=True))

Frequently asked questions

How fresh are the IOCs?

TweetFeed scrapes the source feeds every 15 minutes via cron. The end-to-end pipeline (scrape, dedupe, classify, aggregate, push) takes roughly 17 to 25 seconds, so an IOC tweeted right now lands in today.csv and the API within one cron tick.

Is there a rate limit on the API?

api.tweetfeed.live runs as a Cloudflare Worker on the Free plan (100,000 requests per day). Real traffic sits around 7 to 8K per day, leaving roughly 300x headroom. There is no authentication or per-key throttling.

Can I use TweetFeed data commercially?

Yes. The IOC data is published under CC0 1.0 (no rights reserved). Attribution is appreciated but not required. The website code, branding and logos are not included in CC0.

How do I credit the original researcher?

Every IOC carries a user field (Twitter/X handle) and a tweet field (full URL to the source post). Link back to the tweet whenever you republish or alert on an IOC.

How do I handle false positives?

TweetFeed publishes everything reasonably tagged and does not suppress noise. Use a client-side allowlist (the KQL and SPL recipes show the pattern) and report false positives via the public Featurebase board.

Why is there no authentication?

TweetFeed is open data, side-project scope, CC0-licensed. Adding auth would gate adoption without solving any real abuse. The Cloudflare Worker absorbs spike traffic; if you need a private mirror just clone the GitHub repo.

Is there an MCP server for AI agents?

Yes. mcp.tweetfeed.live exposes 5 tools: query_iocs, check_url, check_ip, check_hash and list_recent_iocs. See the Agents page for the config snippet.

How do I contribute new IOCs?

Post the IOC on Twitter/X with one of the tracked hashtags (full list on the Feeds page) or get added to the curated TweetFeedList. The next cron tick picks it up automatically.

Is there a self-hosted or enterprise version?

No SLA, no enterprise tier - this is a single-maintainer side project. Use TweetFeed as one source of many in your Threat Intelligence pipeline, never as the only source.

Does TweetFeed work in air-gapped environments?

Yes. Clone the data repo (github.com/0xDanielLopez/TweetFeed) inside your perimeter and refresh on whatever cadence your egress policy allows. The CSV layout is documented and stable.
